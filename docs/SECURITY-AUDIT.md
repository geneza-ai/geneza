# Geneza — Security & Correctness Audit

> Multi-agent audit (14 surface reviewers → 2–3 adversarial verifiers per finding → synthesis). 33 findings confirmed of 34 raised; 1 rejected.

**Posture:** needs-work — the cryptographic and transport core is solid, but four high-severity gaps let the exact in-scope adversaries (compromised controller, compromised proxy/relay, any authorized client) exceed the documented blast radius; these are concrete, mostly self-contained fixes rather than redesigns.

**Severity counts (confirmed):** critical 0 · high 5 · medium 9 · low 15 · info 1

## Executive summary

GENEZA's core security architecture is sound and mostly faithfully implemented: the two-tier CA split, Noise IK tunnels with agent-side static-key binding, signed grants, mTLS-gated control plane, single-use rendezvous tokens, and a pinned offline artifact-signing key are all present and working. The system is honestly threat-modeled, and most findings are the auditors holding the code to its own stated guarantees rather than novel attack surface. The dominant problem is not the design but incomplete delivery of the design's own containment promises against the two adversaries the threat model explicitly names — the compromised controller and the compromised web proxy/relay. Four high-severity issues each let one of those in-scope adversaries exceed the blast radius the architecture says it must not: the audit chain is forgeable/truncatable by anyone with file-write access (controller compromise becomes undetectable), the CA root private key is written to and left on the controller at init (controller filesystem compromise becomes fleet-wide identity takeover), the artifact updater has no anti-rollback gate (a compromised controller silently downgrades the fleet to a known-vulnerable worker), client-controlled terminal dimensions OOM the 'sacred' session-host process killing every live PTY on a node, and require_native is unenforceable because the controller trusts a client-asserted field that also fails open on empty. A recurring secondary theme is missing agent-side independent re-validation: the agent backstops the controller on grant signature and node binding but not on TTL ceilings, ForbidDetach reattach, node-cert expiry, or require_native, so several 'the agent bounds a bad controller' guarantees rest entirely on the controller being correct. No memory-corruption, no authentication bypass on the mTLS-gated RPCs, no key-disclosure-over-the-wire, and no tunnel-crypto break were found.

## Systemic themes

- Controller-compromise containment is structurally present but not actually enforced. The architecture's headline promise — that compromising one controller must not become fleet-wide takeover — is undercut at three independent points: the root CA key sits on the controller (identity takeover), the audit chain is unkeyed and host-writable (compromise is undetectable), and the updater has no rollback floor (silent downgrade to vulnerable workers). These are the same root cause: a containment property the design asserts is missing its enforcing mechanism.
- Missing agent-side independent re-validation. The agent correctly backstops the controller on grant signature, node binding, and Noise-key binding, but NOT on the bounds the threat model says it should: no upper ceiling on grant ExpiresAt or MaxSessionTTL (grant.go, offer.go), no ForbidDetach check on reattach (agentd/session.go:186 hardcodes detachable=true), no node-cert-expiry gate on new offers, and no require_native re-check. Every 'the agent bounds a bad/compromised controller' guarantee therefore rests solely on the controller being correct.
- Trusting client-supplied values instead of deriving them from the authenticated principal/channel. require_native trusts req.client_path; terminal dimensions trust client cols/rows unbounded; the empty-string carve-out in policy.go fails open. Authority is being read from request payload rather than from verified identity or hard server-side limits.
- Unbounded resource allocation from a single request, with no global ceiling. Recurs across surfaces: terminal grid (session-host OOM), JWKS refetch-per-unknown-kid (controller+IdP DoS), relay pre-hello connections (no LimitListener/slowloris cap), session records never reaped (unbounded bbolt growth), and TOCTOU on max_sessions/max_detached. The §13 'bounded state / bounded resource exhaustion' guarantee is asserted but not consistently backed by caps.
- Append-only / single-use / monotonic invariants are claimed but not enforced. The audit cast file is O_TRUNC-overwritable by the producing node, session state can move backward from Ended to Active, the single-use rendezvous token becomes re-matchable after MatchTTL while the splice is still live, and the updater has no version monotonicity. Each is a stated one-way property that the code lets run backward.
- Crash/error robustness gaps turn benign faults into control-plane outages. A torn final audit line bricks controller startup; a malformed time_window day or a non-32-byte pubkey panics with no recovery interceptor. Fail-closed is good, but here it fails closed on availability for operator typos and ordinary crashes rather than degrading gracefully.
- TLS/trust-anchor edges fail open. The update HTTP client defaults to InsecureSkipVerify when ca_roots_file is unset, the client re-TOFUs when ca.pem is missing despite a recorded pin, and controller-pushed CA rotation is adopted with no overlap/continuity check — three places where a trust anchor can be silently replaced. Substring-matching on error messages ('pin mismatch') instead of typed sentinel errors is the proximate cause of the TOFU bypass.
- Documented §9 CA-rotation-with-overlap is not implemented; rotation is a hard cutover provided only by manual file curation, which the architecture itself flags as the classic way to lock yourself out of your fleet.

## Remediation order

1. Clamp terminal cols/rows to a hard maximum (<=1000x1000) in startSession and handleResize (and pty.Setsize) — highest risk-reduction per effort: a one-line-per-site bound that removes a single-frame total-node DoS reachable by any authorized client. Trivial, no protocol change.
2. Make require_native enforceable: stop trusting req.client_path, derive ClientPath from the authenticated identity server-side, and make policy.go fail closed (require =="native", drop the empty-string carve-out). Closes the high-severity web-proxy containment bypass; the policy.go fail-closed half is a one-condition change usable immediately.
3. Add an anti-rollback floor to the updater: persist a monotonic version high-water-mark in bootstrap State and reject non-newer manifests (mirror the existing VerifyClusterConfig ConfigVersion check). Pattern already exists in the codebase, so this is low-risk and stops silent fleet downgrade to vulnerable workers.
4. Move the CA root key off the controller: split root generation into an offline/HSM step so InitDataDir provisions only the issuing key + certs and root-ca.key never persists in the controller data_dir. Higher operational effort (changes the provisioning ceremony) but removes the single-host-to-fleet-takeover path; sequence after the quicker high-sev fixes.
5. Anchor the audit chain: HMAC/sign records or periodic checkpoints with a key the controller-compromise attacker doesn't control, add a monotonic sequence number to detect tail truncation, cross-check a persisted count/last-hash from the bbolt store, and ship to an off-box append-only sink. Larger change but it is the only fix that restores tamper-evidence as a real detective control.
6. Harden the controller control-plane DoS surfaces together (medium cluster): JWKS minimum-refetch interval + key-count cap + fetch outside the mutex; relay LimitListener/MaxConns + per-IP cap; session-record reaper for terminal/stuck-pending records; atomic check-and-reserve for max_sessions/max_detached plus a reaper backstop for max_sessions.
7. Add the missing agent-side independent bounds (medium/low cluster, defense-in-depth): clamp grant ExpiresAt/MaxSessionTTL to a local ceiling and always arm the close timer; pass detachable=grant.AllowDetach and enforce ForbidDetach on reattach (terminate rather than re-detach to drain pre-existing sessions); reject offers once the local node cert has expired; re-check the require_native path at the agent.
8. Fix the fail-open trust-anchor edges: require ca_roots_file (or pin the cert) instead of defaulting to InsecureSkipVerify; fail closed in establishCATrust when a pin is on record but ca.pem is missing (match a typed error, not substring); require overlap/continuity and show the full fingerprint on controller-pushed CA rotation.
9. Robustness/availability hardening: tolerate one torn final audit line on open (truncate to last good record) so a crash doesn't brick startup; validate time_window.days at Parse time and add a gRPC panic-recovery interceptor; add the len(pub)==32 guard to VerifyOne; add a node-still-enrolled check to RenewCert; switch UploadRecording to O_EXCL/terminal-state-only; make session-state transitions monotonic; keep the rendezvous token non-rematchable for the full splice lifetime; health-gate the rolled-back worker.
10. Implement the §9 CA-rotation-with-overlap tooling and the desired-version OS/arch fix (linux/amd64 hardcode breaks self-update for macOS and linux/arm64) — conformance/feature gaps to schedule once the security-critical items above are landed.

## Notable strengths

- No findings against the tunnel cryptography itself: Noise IK is used correctly, and the agent independently verifies remoteStatic==grant.ClientNoisePub (offer.go:149-151), so even a compromised relay that hijacks rendezvous cannot inject or MITM the encrypted payload — the worst token/relay findings are bounded to rendezvous DoS/metadata precisely because this binding holds.
- The mTLS-gated control plane held up under review: no authentication-bypass finding on any cert-gated RPC. The single deliberately unauthenticated endpoint (UserAPI/Login) is a conscious design choice, and the findings there are amplification/timing (JWKS refetch, bcrypt enumeration), not auth bypass.
- Defense-in-depth that worked is visible in the findings' own bounding language: NodeControl/Stream DOES re-check node enrollment (the RenewCert gap is called out as an asymmetry against a correct sibling), resolveAttach DOES enforce rec.User==caller (which confines the session-reactivation issue to info severity), and the cluster-config path DOES enforce ConfigVersion monotonicity (the model the updater should copy). The right patterns exist in-tree.
- The artifact-trust anchor is genuinely robust where it matters most: binaries are validated against a pinned offline signing key, so even with the TLS-verification-disabled and rollback findings, an attacker cannot inject an arbitrary/forged binary — they are limited to selecting among legitimately-signed builds. The signing key is correctly kept off the controller (only the rollback freshness check is missing).
- Fail-closed instincts are pervasive and mostly correct: the install path hard-rejects wrong-platform binaries, grant.Validate rejects expired/future-dated grants within tolerance, and the existing audit tamper tests do catch in-place edits and mid-file deletion. Several findings are explicitly 'not attacker-reachable today' (e.g. VerifyOne panic) — caught as latent footguns, not live holes.
- Severity discipline by the reviewers reflects a system that is honestly bounded: the bulk of issues are low/info because cross-tenant and cross-node isolation generally holds (e.g. the session-reactivation and capacity-TOCTOU issues stay within a node's existing authority), indicating the core isolation boundaries are sound even where individual guardrails are loose.
- High reviewer agreement (every high-severity finding at 3/3, the rest at 2/2) and the absence of any disputed or speculative finding indicate the codebase is legible and the threat model is concrete enough to audit against — itself a sign of disciplined security engineering.

## Confirmed findings

### 1. [HIGH] Artifact update path has no anti-rollback/freshness check: a compromised controller can replay an old signed manifest to downgrade the fleet to a known-vulnerable worker

- **Category:** security  ·  **Location:** `/root/geneza/cmd/geneza-bootstrap/main.go:317-344 (reconcile), with /root/geneza/internal/update/install.go:46-72 and /root/geneza/internal/types/manifest.go:16-24`  ·  **Verifier agreement:** 3/3
- **What:** The bootstrap accepts ANY validly-signed manifest whose Version differs from the running version. The only gate before install is `d.Version == b.current` (main.go:317) and the clearable bad-list (main.go:321). The manifest carries a `CreatedAt` (manifest.go:23) but it is never read, and no persisted version high-water-mark exists. By contrast, the CA/cluster-config push path DOES defend against exactly this: `VerifyClusterConfig` rejects any config whose `ConfigVersion` is below the current one (`internal/types/cluster.go:59-61`, "cluster config rollback"). The binary path omits the equivalent defense. Because the offline signing key signs every published build forever (no expiry/revocation in the manifest), every old build's manifest remains a valid, installable artifact.
- **Impact:** Adversary = compromised controller (explicitly in scope, ARCHITECTURE.md §12 "Compromised controller"). The controller sets desired version via SetDesiredVersion (adminapi.go:196) with no monotonicity check and serves the stored old signed manifest + blob. The bootstrap installs it, the old (real) binary passes its own health gate, the swap commits, and the fleet is silently downgraded to a previously-shipped worker that may contain a since-patched RCE/auth-bypass. This re-opens fixed vulnerabilities fleet-wide using only controller control — defeating threat-model guarantee 4 ("malicious binary push is defeated by ... staged rollout") since a downgrade is a malicious push of a still-validly-signed build.
- **Fix:** Persist a monotonic version high-water-mark in the bootstrap State (mirroring ConfigVersion in cluster config) and reject any manifest whose version (or signed CreatedAt) is not strictly newer than the high-water-mark, except via an explicit, separately-authorized operator rollback flow. At minimum, enforce a bounded manifest freshness window using the signed CreatedAt so stale manifests cannot be replayed indefinitely.

### 2. [HIGH] Audit chain is an unkeyed self-hash with no external anchor: forgeable and tail-truncatable by anyone with file-write access (the controller-compromise adversary)

- **Category:** security  ·  **Location:** `internal/controller/audit.go:31-39, 144-192`  ·  **Verifier agreement:** 3/3
- **What:** auditHash() is a plain SHA-256 over the record JSON with Prev embedded; the chain link is hash[i] = SHA256(record[i] with Hash=""), and the genesis record has Prev="". There is no key (no HMAC, no ed25519 signature with the grant key), no persisted record count, no signed head/checkpoint, and no off-box anchor (no S3 object-lock / SIEM as §13 requires). scanAuditFile only checks that each record's Prev equals the previous record's Hash and that each Hash recomputes. Consequences proven empirically: (1) TAIL TRUNCATION is undetectable — deleting the most recent N records (e.g. the very events recording an attacker's session/artifact-publish/policy-reload) leaves a perfectly valid chain that VerifyAuditFile returns err=nil for. (2) WHOLE-CHAIN RE-FORGE is trivial — an attacker who can write audit.jsonl can drop/alter any records and recompute every Hash from a new genesis, producing an arbitrary forged history that verifies clean. The TestAuditTamperDetection/TestAuditTruncationDetection tests only exercise in-place single-field edits and mid-file deletion, which do break the chain; they do not cover tail-truncation or full re-chaining.
- **Impact:** A compromised controller (the exact adversary the threat model must bound) can erase or rewrite its own audit trail and the verifier (CLI audit-verify, QueryAudit ChainOk) will still report the chain as intact. Defeats the §13 'tamper-evident audit' guarantee and the threat-model promise that controller compromise is detectable/recorded.
- **Fix:** Anchor the chain cryptographically and against truncation: sign each record (or a periodic checkpoint of {count,last_hash,ts}) with the controller grant key or a dedicated audit key, and have verify require a valid signature on the head plus a monotonically increasing sequence number so missing tail records are detected. Persist the expected record count / last hash in the bbolt store (separate trust domain) and cross-check on open and query. For true tamper-evidence against host compromise, ship records to an append-only off-box sink (S3 object-lock / SIEM) as §13 specifies.

### 3. [HIGH] require_native is unenforceable: client-supplied client_path is trusted and empty value fails open

- **Category:** security  ·  **Location:** `internal/controller/broker.go:133, 174 (with internal/policy/policy.go:211)`  ·  **Verifier agreement:** 3/3
- **What:** The broker copies the caller-supplied request field req.GetClientPath() verbatim into the policy Input (broker.go:133) and into the signed grant (broker.go:174). The policy rule guard is `if r.RequireNative && in.ClientPath != "" && in.ClientPath != "native" { return false }` (policy.go:211). There is exactly one user cert kind (ca.go:40-41, KindUser); the controller has no server-side way to distinguish a native-CLI caller from the web-proxy caller — the only signal is the self-asserted client_path in the gRPC request. The native CLI happens to set client_path="native" (client/session.go:87), but nothing verifies that. Any caller holding a valid user cert (a compromised web proxy, which the threat model explicitly treats as semi-trusted and bounds via 'policy reserves sensitive targets for native', or anyone with a stolen short-TTL user cert) can reach a require_native-restricted target two ways: (a) set client_path="native", or (b) omit client_path entirely, since the empty-string carve-out in policy.go:211 makes require_native rules MATCH when ClientPath=="". The agent does not re-check this: grant.Validate (types/grant.go:54-88) and offer.go:40 never inspect ClientPath/require_native, so the controller is the sole enforcement point and it relies on an attacker-controlled value.
- **Impact:** A compromised web session proxy (inside the trust boundary but explicitly meant to be bounded by 'sensitive targets restricted to native client by policy', ARCHITECTURE.md:169/271), or any holder of a stolen user cert, can broker plaintext-visible web-path sessions to targets that policy intended to reserve for the true-E2E native client. This defeats the threat-model guarantee that a compromised proxy cannot exceed policy for sensitive targets.
- **Fix:** Do not trust the request field. Derive client_path server-side from the authenticated principal/channel: give the web proxy a distinct cert kind/role (or a dedicated RPC) and set ClientPath authoritatively in the broker before evaluation, ignoring req.GetClientPath(). Additionally make policy.go:211 fail closed: a require_native rule must NOT match unless ClientPath == "native" (drop the `in.ClientPath != ""` exception). Also have the agent enforce require_native by including the resolved path in the signed grant and re-checking it.

### 4. [HIGH] InitDataDir provisions the CA ROOT private key onto the controller host, violating 'controller holds neither the CA root key'

- **Category:** security  ·  **Location:** `internal/controller/init.go:38`  ·  **Verifier agreement:** 3/3
- **What:** InitDataDir calls ca.Init(cfg.CADir(), cfg.ClusterName) with cfg.CADir() = <data_dir>/ca. ca.Init (internal/ca/ca.go:103-178) generates BOTH the root CA key and the issuing CA key and writes root-ca.key (0600) and issuing-ca.key (0600) into that same controller data_dir, then returns. The controller runtime (ca.Load, ca.go:182-183) only reads issuing-ca.key, but root-ca.key is left on the controller's disk indefinitely; nothing moves it offline or removes it. The root certificate is a self-signed CA with MaxPathLen:1 that signs the issuing CA, so possession of root-ca.key lets an attacker mint a brand-new issuing CA and any node/user identity in the cluster.
- **Impact:** Filesystem compromise of the controller yields the CA ROOT key = fleet-wide identity takeover, directly contradicting threat-model guarantee #2 ('the controller holds neither the CA root key nor the artifact signing key') and §2 ('CA signing key ... Kept offline / in HSM / KMS, never on the controller'). The very property that is supposed to bound a controller compromise is broken at provisioning time.
- **Fix:** Generate the root key on a separate offline/HSM step (or via the ca package writing it to a distinct, removable location), and have InitDataDir provision only the issuing CA key + issuing/root certs on the controller. After issuing-ca creation, explicitly require root-ca.key to be moved off-box (or never written to the controller data_dir at all); document/enforce that the controller data_dir contains no root private key.

### 5. [HIGH] Unbounded terminal grid allocation from client-controlled cols/rows (Resize and Create) OOMs the entire session host

- **Category:** security  ·  **Location:** `internal/sessionhost/session.go:355-376 (handleResize); also sessionhost.go:248-254,314`  ·  **Verifier agreement:** 3/3
- **What:** handleResize accepts cols/rows as uint32 from the attached client with no upper bound and calls s.vt.Resize(int(cols), int(rows)). vt10x's resize() (state.go:304-348) eagerly allocates 2*rows lines, each a slice of cols Glyphs (~24 bytes each). A single Resize{Cols:100000,Rows:100000} forces an allocation of ~2*10^10 Glyphs (hundreds of GB) and OOM-kills the process. The same unbounded path exists at Create time: startSession passes req.Cols/req.Rows straight into vt10x.New(WithSize(int(cols),int(rows))) (sessionhost.go:314) and stores them, and renderSnapshot does b.Grow(cols*rows+256) (screen.go:109). pty.Setsize truncates to uint16 so the kernel PTY size is unaffected, but the in-process vt model and snapshot buffer are not bounded.
- **Impact:** Any single authorized client of one session (or a compromised web proxy / controller bridging the Attach stream) can send one Resize frame and crash the session host process. Per ARCHITECTURE §8/§13 this process is 'sacred' and hosts ALL persistent PTYs on the node, so one frame kills every live session on the host. Violates threat-model guarantee 5 (agent resource exhaustion must be bounded).
- **Fix:** Clamp cols/rows to a sane maximum (e.g. <=1000x1000, matching real terminals) in both handleResize and startSession before touching the vt or recording the size; reject or silently cap oversized values. Also bound renderSnapshot's Grow by the clamped size.

### 6. [MEDIUM] Controller TLS verification is disabled by default (InsecureSkipVerify) when ca_roots_file is unset, leaving the desired-version channel tamperable

- **Category:** security  ·  **Location:** `/root/geneza/internal/update/httpclient.go:45-51`  ·  **Verifier agreement:** 2/2
- **What:** NewHTTPClient sets `tlsCfg.InsecureSkipVerify = true` whenever `caRootsFile` is empty (httpclient.go:46), logging a warning but proceeding. The bootstrap config makes ca_roots_file optional (config.go has no default and loadConfig does not require it), so the default deployment runs with controller TLS completely unverified. While binary integrity is independently protected by the pinned offline signing key (so an injected blob is rejected on hash mismatch), the desired-version poll response (FetchDesired) and the choice of WHICH signed manifest to serve are not signature-bound to the request — any on-path attacker (not just the controller) can select which validly-signed manifest the node installs, including an older one (compounds the rollback finding), and can observe node identity/version metadata in cleartext.
- **Impact:** An on-path MITM (relay operator, intermediate network, the documented threat surface for untrusted infra) can, without compromising the controller, force the node to install an attacker-chosen but validly-signed (e.g. older/vulnerable) worker version and can map node IDs to versions. This materially weakens the "untrusted infrastructure" posture for the update channel.
- **Fix:** Require ca_roots_file (fail closed at loadConfig if absent), or pin the controller certificate; do not silently fall back to InsecureSkipVerify. If a bootstrap-without-roots mode must exist for first-boot, gate it behind an explicit config flag rather than the absence of a value.

### 7. [MEDIUM] TOFU CA pin is bypassed when ca.pem is missing — silently re-TOFUs instead of failing closed

- **Category:** security  ·  **Location:** `cmd/geneza/login.go:253-282`  ·  **Verifier agreement:** 2/2
- **What:** establishCATrust enforces the recorded pin (profile.CASHA256) only via st.LoadCA(prevPin), and only treats an actual content mismatch as fatal: `else if strings.Contains(err.Error(), "pin mismatch")`. When the profile exists with a non-empty CASHA256 but ca.pem is absent, LoadCA returns "no CA bundle for this profile (run 'geneza login')" (profile.go:99-101) — which does NOT contain "pin mismatch". The code therefore falls through (lines 262-282) to FetchCARootsTOFU and re-pins whatever the controller now serves, with only a 'first contact' banner. The recorded pin is never consulted in this branch. This contradicts the guarantee the code itself prints ('any future mismatch fails closed') and ARCHITECTURE §5/§12 (stolen/short-lived creds, fail-closed trust). An adversary with local file access who deletes only ca.pem (leaving profile.json), combined with a network MITM on the next login's InsecureSkipVerify TOFU fetch (httpapi.go:82-99), gets their CA bundle pinned with no mismatch error — defeating TOFU pinning.
- **Impact:** A pinned trust anchor on record (profile.CASHA256) can be silently replaced. An attacker who can delete one file and MITM the next login installs an attacker-controlled CA, after which all controller/relay TLS validates against the attacker root, enabling full session MITM. Defeats the stated fail-closed pinning guarantee.
- **Fix:** Before falling through to TOFU, fail closed when a pin is on record: if prevPin != "", any LoadCA error (including missing file) must abort with an instruction to re-run with --ca-file or to delete the whole profile dir. Only TOFU when prevPin == "" (genuine first use). Match on a typed/sentinel error rather than substring matching on the message.

### 8. [MEDIUM] Reattach path ignores ForbidDetach agent policy and re-detaches sessions unconditionally

- **Category:** security  ·  **Location:** `internal/agentd/session.go:186 (serveAttach); enforcement gap relative to offer.go:46 and sessionhost.go:213-215`  ·  **Verifier agreement:** 2/2
- **What:** On the attach action, serveAttach calls bridgeAndFinish(..., detachable=true, ...) with the detachable flag HARDCODED to true (line 186), ignoring both grant.AllowDetach and the local agent policy pol.ForbidDetach. EvaluateOffer (offer.go:46) only rejects when grant.AllowDetach && pol.ForbidDetach, but the controller routinely issues attach grants with AllowDetach=false (e.g. when the client did not pass WantDetachable, or after policy tightened), so the offer is accepted and the worker still re-detaches on client-gone. The session host only blocks NEW detachable sessions in Create (sessionhost.go:213) — it never re-evaluates ForbidDetach for an existing session being reattached. Each successful attach also resets s.detachedSince to zero (sessionhost.go:239), restarting the detached-TTL reaper clock. Net effect: once a detachable session exists, an operator who pushes ForbidDetach to that node to kill the 'long-lived detached root shell' persistence vector (explicitly called out in ARCHITECTURE.md §8) cannot actually terminate it — a holder of short-TTL attach grants can reattach and re-detach indefinitely, keeping the detached shell alive forever and defeating both ForbidDetach and detached-TTL reaping.
- **Impact:** Defeats the ForbidDetach guardrail (threat-model item 5, detached-session abuse). An attacker or user with valid short-TTL credentials keeps a pre-existing detached shell alive on a node where policy now forbids detachment, evading the documented persistence-containment control and TTL reaping.
- **Fix:** Pass detachable = grant.AllowDetach (and reject the attach in EvaluateOffer/serveAttach when pol.ForbidDetach regardless of grant.AllowDetach). When ForbidDetach is in force, a reattach whose client goes away should terminate the session (Kill) rather than re-detach it, so the guardrail can actually drain pre-existing detached sessions.

### 9. [MEDIUM] Root CA private key is generated on and left on the controller host, violating the offline-root threat-model guarantee

- **Category:** security  ·  **Location:** `internal/ca/ca.go:95-179 (Init); written at 167 root-ca.key; controller init at internal/controller/init.go:38; CADir at internal/controller/config.go:204`  ·  **Verifier agreement:** 2/2
- **What:** ca.Init() (invoked by `geneza-controller init` via controller.InitDataDir -> ca.Init(cfg.CADir(),...)) generates the ROOT CA private key in-process on the controller and writes it to root-ca.key (mode 0600) inside cfg.CADir() = DataDir/ca — the very directory the running controller also uses (ca.Load reads issuing-ca.key from the same dir, internal/controller/server.go:61). Nothing in the code moves, encrypts, deletes, or even warns that root-ca.key must be taken offline; it persists on the controller's disk indefinitely. ARCHITECTURE.md §2 and §9 and threat-model guarantee #2 require that the CA root key be 'kept offline / in HSM / KMS, never on the controller' and that 'the controller holds neither the CA root key nor the artifact signing key.' The two-tier split (offline root, online issuing) is implemented structurally, but the root key is co-located with the controller, defeating the containment the split is supposed to provide.
- **Impact:** An adversary who compromises the controller host (the explicit 'compromised controller' threat actor) can read root-ca.key and mint an entirely new issuing CA, then sign node/user/admin leaves of their choosing that every agent and client in the fleet trusts (ca-roots.pem anchors the root). This converts a single-host controller compromise into fleet-wide identity takeover and defeats CA rotation/short-TTL containment — exactly the blast-radius the design says must not be exceeded.
- **Fix:** Split root-CA generation out of the controller provisioning path: generate the root key on a separate offline workstation/HSM and have the controller init consume only a root cert + a freshly issued issuing keypair (or a CSR-signing ceremony). At minimum, after producing issuing-ca.{key,crt} and ca-roots.pem, Init should NOT leave root-ca.key in CADir on the controller — write it to a distinct, operator-specified offline path and emit a hard warning, or wrap the root key under KMS/Vault transit so the on-disk artifact is non-usable by a controller-host attacker.

### 10. [MEDIUM] Desired-version endpoint hardcodes linux/amd64, silently breaking the self-update reconcile loop for macOS and linux/arm64 nodes

- **Category:** conformance  ·  **Location:** `internal/controller/httpapi.go:88`  ·  **Verifier agreement:** 2/2
- **What:** GET /v1/updates/desired resolves the version via resolveDesired() and then always looks up ManifestKey("geneza-agent", "linux", "amd64", desired), ignoring the requesting node's OS/arch. The bootstrap sends only ?node=&current= (internal/update/httpclient.go:65-69) and never communicates its platform, while PublishArtifact stores manifests keyed by the manifest's real os/arch (adminapi.go:178, ManifestKey(m.Product,m.OS,m.Arch,m.Version)). So a darwin or linux/arm64 node either (a) receives a linux/amd64 SignedManifest, which the installer hard-rejects on the m.OS/m.Arch != ins.OS/ins.Arch check (internal/update/install.go:61-63), or (b) gets a 404 because no geneza-agent/linux/amd64/<version> manifest was ever published. A panic-recovery interceptor on the gRPC server would also contain the blast radius.

### 28. [LOW] Single-use token guarantee broken for long sessions: `used` entry evicted after MatchTTL, not at splice end

- **Category:** security  ·  **Location:** `internal/relay/relay.go:247-270`  ·  **Verifier agreement:** 2/2
- **What:** On a match, the relay deletes the token from pending, inserts it into `used`, and schedules its removal via time.AfterFunc(MatchTTL, ...) (relay.go:250-257), then runs splice() synchronously for the entire session. The eviction timer is keyed to MatchTTL (default 60s, defaults.go:50), but a splice lives until idle for IdleTimeout (default 10 min, refreshed on every read in copyHalf, relay.go:372) — i.e. potentially hours for an active shell. After MatchTTL elapses, the token is in NEITHER pending NOR used while the original splice is still live. A new hello carrying that same token then falls through to the `default` branch (relay.go:272-291) and is parked as a fresh waiting slot; a second opposite-role hello with the same token splices a brand-new rendezvous. The inline comment (relay.go:248-249) states the intent is 'so a replayed hello cannot pair a second time', but that intent holds only for MatchTTL, not the session lifetime. Anyone who learns the token can exploit this — most relevantly a COMPROMISED RELAY, which parses the full token out of the hello frame (relay.go:212-213) even though it only logs the prefix.
- **Impact:** Breaks the documented single-use token invariant (config.go:36 'one-time token'; wire/frame.go:62 'Tokens are single-use'). A token-holder (e.g. a compromised relay) can, 60s into a long session, stand up a second unauthorized rendezvous on the same token and lure a re-dialing endpoint, enabling targeted rendezvous denial/confusion. The Noise IK layer + the agent's remoteStatic==grant.ClientNoisePub check (offer.go:149-151) prevent payload injection/MITM, so impact is bounded to rendezvous-hijack/DoS and metadata, not tunnel compromise — hence medium, not high.
- **Fix:** Keep the token in `used` until the splice ends (e.g. add it back / refresh on each copyHalf cycle, or remove it in splice's deferred teardown), and only then start a short grace timer; or set the used-eviction delay to at least MaxSessionTTL/IdleTimeout rather than MatchTTL. The token must remain non-rematchable for the entire lifetime of the spliced connection.

### 29. [LOW] Non-atomic capacity check lets concurrent Create exceed max_sessions / max_detached

- **Category:** bug  ·  **Location:** `internal/sessionhost/sessionhost.go:222 and 226 (check) vs 372 and 375 (increment)`  ·  **Verifier agreement:** 2/2
- **What:** host.Create reads the live counters with `h.detachedN.Load() >= maxDetached` (line 222) and `h.activeN.Load() >= maxSessions` (line 226), but the corresponding `h.activeN.Add(1)` / `h.detachedN.Add(1)` happen later in startSession (lines 372/375) with no lock held across the read-modify span. gRPC dispatches each Create in its own goroutine, so N concurrent Create RPCs arriving while the counter is at limit-1 can all observe `< max` and all proceed to increment, overshooting the cap. The same Load-then-later-Add pattern recurs in session.clientGone (session.go:283) and detachRequest (session.go:311) which read detachedN to decide keep-vs-reap while other sessions concurrently increment it. The reaper eventually trims over-cap detached sessions, but max_sessions itself has no reaper backstop, so active-session overshoot persists for the life of those sessions.
- **Impact:** A control plane that brokers many grants concurrently (notably a compromised controller, which can broker/replay multiple valid grants in parallel) can drive the node above its configured max_sessions guardrail by roughly the number of simultaneously in-flight Create calls, weakening the bounded-resource-exhaustion guarantee. Bounded overshoot, not unbounded.
- **Fix:** Make the check-and-reserve atomic: either hold h.mu across the capacity test and the counter increment (the sessions map is already guarded by h.mu and updated in startSession), or replace the Load/Add pair with a CAS loop that reserves a slot before startSession and releases it on failure. Add an equivalent atomic reservation for the detached counter.

### 30. [LOW] Agent does not cap grant ExpiresAt / MaxSessionTTL upper bound (no independent long-lived-grant ceiling)

- **Category:** conformance  ·  **Location:** `internal/types/grant.go:64-69`  ·  **Verifier agreement:** 2/2
- **What:** SessionGrant.Validate enforces a lower clock-skew bound and that now is not after ExpiresAt, but imposes no upper bound on how far in the future ExpiresAt may be, and offer.go:170 honors grant.MaxSessionTTL verbatim with no agent-side ceiling. Today the controller sets ExpiresAt = now + grantTTL (default 2m, defaults.go:41) and MaxSessionTTL <= DefaultMaxSessionTTL (12h), so legitimate grants are short. However, the agent re-validation — which the threat model positions as the independent check that bounds a compromised/misconfigured controller (guarantee 3: 'short-TTL ... no long-lived secrets in the data path'; guarantee 5: 'clock skew ... is bounded') — provides no ceiling of its own. A controller that is compromised or misconfigured with a huge grant_ttl can mint a grant whose rendezvous window (and thus the credential's validity) is arbitrarily long, and the agent accepts it. Because the controller holds the grant signing key this is partly inherent to the design, but an agent-side maximum (e.g. reject ExpiresAt more than N minutes after IssuedAt, and clamp MaxSessionTTL) is exactly the kind of independent bound the spec calls for and would shrink the blast radius of a controller compromise.
- **Impact:** A compromised or fat-fingered controller can issue a grant with a far-future expiry/session TTL; the agent enforces no independent ceiling, so the short-lived-credential and bounded-skew guarantees rest entirely on the controller being correct rather than on agent-side re-validation.
- **Fix:** In Validate, reject grants whose ExpiresAt is more than a fixed maximum (e.g. defaults.GrantTTL + small slack, or a configurable agent ceiling) after IssuedAt, and clamp/validate MaxSessionTTL against an agent-side maximum before use in offer.go.

### 31. [LOW] VerifyOne does not validate public-key length before ed25519.Verify (panic on non-32-byte key)

- **Category:** bug  ·  **Location:** `internal/types/signed.go:69-78`  ·  **Verifier agreement:** 2/2
- **What:** VerifyOne calls ed25519.Verify(pub, ...) directly without first checking len(pub) == ed25519.PublicKeySize, unlike the sibling Verify (signed.go:52) which guards both pub and sig lengths. The Go standard library ed25519.Verify panics (not returns false) when the public key is not exactly 32 bytes (confirmed empirically: 'ed25519: bad public key length: 10'). A wrong-length signature returns false safely, but a wrong-length public key panics. All three current callers (install.go:55 ins.Pub guarded at :47; adminapi.go:110 s.artifactPub from LoadPublicKeyPEM; geneza-sign main.go:166 pub from LoadPublicKeyPEM) source pub from locally-loaded PEM that is length-validated, so this is not attacker-reachable today. It is a fail-open-to-panic asymmetry versus Verify and a footgun for any future caller that passes an unvalidated key.
- **Impact:** No remote adversary today (pub is always operator-controlled and length-checked upstream). A future caller passing an unvalidated/attacker-influenced public key into the bootstrap's offline-verification path would get a process panic (local DoS) instead of a clean ErrBadSignature, weakening the fail-closed posture of the binary-trust anchor.
- **Fix:** Add the same guard as Verify at the top of VerifyOne: if len(pub) != ed25519.PublicKeySize || len(s.Sig) != ed25519.SignatureSize { return ErrBadSignature }.

### 32. [LOW] Data-path frame read cap (1 MiB) is 32x the sender's real frame size, letting a blind relay pin/amplify receiver memory

- **Category:** security  ·  **Location:** `internal/wire/frame.go:12,32-36 (cap); internal/tunnel/noise.go:27,59-67 (read path)`  ·  **Verifier agreement:** 2/2
- **What:** wire.ReadFrame accepts any length-prefixed frame up to MaxFrame = 1<<20 (1 MiB) and immediately does make([]byte, n) for the declared length before reading the body. On the established tunnel data path the sender (tunnel.Conn.Write) chunks plaintext at MaxPlaintext = 32 KiB, so every legitimate post-handshake ciphertext frame is only ~32 KiB+16 bytes. The 4-byte length prefix is written in cleartext outside the Noise AEAD (WriteFrame writes the header, then the ciphertext; the header is not part of the AEAD associated data), so a relay that blindly splices the byte stream can rewrite the length prefix to up to 1 MiB. After the handshake, both endpoints clear their read deadline (client/session.go:147 SetReadDeadline(time.Time{}), agentd/offer.go:159 SetDeadline(time.Time{})), so wire.ReadFrame on the data path has no timeout. A relay can therefore send a header declaring 1 MiB and then stall, causing the receiver to allocate 1 MiB and block in io.ReadFull indefinitely (per spliced session). The post-handshake Decrypt path also never re-enforces Noise's 65535-byte MaxMsgLen (that check exists only in the handshake WriteMessage/ReadMessage), so nothing downstream bounds the frame to the real 32 KiB working size either.
- **Impact:** A compromised/blind relay -- explicitly an untrusted component in the threat model (sec 12: 'Sees only ciphertext + traffic metadata; cannot decrypt or inject') -- can amplify each idle/stalled spliced session's receiver memory footprint ~32x (32 KiB expected vs 1 MiB allocatable) and pin it with no read-deadline bound, across many sessions, weakening the 'relays safe on untrusted infra' guarantee and the agent resource-exhaustion bound (sec 13).
- **Fix:** Introduce a data-path-specific max frame size of MaxPlaintext + AEAD overhead (e.g. 32*1024 + 16) and have tunnel.Conn.Read enforce it (either by passing a max to a ReadFrame variant or validating the length before allocating). Keep the larger MaxFrame only for the relay control/hello frames if needed. Optionally apply an idle read deadline on the data path so a relay-stalled ReadFull cannot pin an allocation indefinitely.

### 33. [INFO] A node can re-activate its own ended session via established/attached events (no state-transition validation)

- **Category:** bug  ·  **Location:** `/root/geneza/internal/controller/nodecontrol.go:141-160`  ·  **Verifier agreement:** 2/2
- **What:** handleSessionEvent applies state transitions unconditionally for the calling node: 'established' and 'attached' set State=SessionActive (lines 142-147) and may set HostSessionID from the event (lines 131-133) with no check on the prior state. A node that owns a session can thus move it from SessionEnded back to SessionActive and rewrite HostSessionID, after which resolveAttach (broker.go:253) would again allow that session to be attached (state is active and HostSessionID is set). The NodeID==nodeID guard (line 125) is correctly evaluated first, so this is strictly confined to the node's own sessions, and resolveAttach still requires rec.User==caller, so it cannot pull another user/tenant onto the box. Impact is therefore contained within the node's existing blast radius (it already owns those PTYs and plaintext).
- **Impact:** No cross-tenant or cross-node effect: only a node's own sessions for their original owning user are affected, and that node already controls those PTYs. The state machine is simply not monotonic, which can confuse fleet/session dashboards but does not exceed the node's authority.
- **Fix:** Make session-state transitions monotonic on the controller: ignore 'established'/'attached' events once a session is in the terminal SessionEnded state, and reject HostSessionID rewrites once it has been set to a different value.
